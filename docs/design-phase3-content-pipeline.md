# Phase 3: 内容ブラッシュアップ — 方向確定（Fable 検証 2026-07-12）

F1〜F6・F8 実装後のユーザー方針案を Fable が検証し、修正を加えて確定した記録。
**方向は確定、詳細設計は次の Fable 設計セッションで確定してから実装する。**

前提: 選定基盤の大幅な作り直しは終了。2-3c hotfix（`docs/design-2-3c-hotfix.md`）を
閉じた後、本文書の方向で生成パイプラインを再設計する。

## 確定事項

### D1. analysis_feature の採用 — ただし台帳パイプライン稼働後に解禁

`抓特务` 型の業界分析・特集記事は一律除外せず、独立記事タイプ `analysis_feature` として扱う。
ただし現行の「原文5000字を直接プロンプトへ入れて書かせる」構造では
「原文の構成・論点順をなぞらない」「全文翻訳に近い要約にしない」を機械的に担保できないため、
**解禁は D3 の事実台帳パイプライン稼働後（Phase 3c）**。それまで F5 の skip を維持する。

掲載条件（すべて必須）:

- 元媒体の分析であることを明示し、元記事の分析とこちらの判断を明確に分ける
- 原文の構成・論点順・比喩・表現をなぞらない。全文翻訳に近い要約にしない
- 「秘書としてのコメント」「今後見るべき点」「日本語圏との接続」を**独立した JSON フィールド**として
  構造化し、**最低2つ非空**であることを機械検査する
- 1回の出力につき analysis_feature は **1本まで**（翻案リスクの露出面積を抑える）

必要な改訂（Phase 3c 実装時）:

- `docs/editorial-character.md` の「媒体の業界分析・特集・深度取材記事も、それ自体を
  1本のニュースとして扱わない」（注意する軸）を analysis_feature 条件付き許可へ書き換える
- `docs/design-2-3b-quality-fixes.md` F5 の skip 指示（文言確定済み）を明示的に supersede する

### D2. 秘書の人格を出力の中心にする — 定義は editorial-character.md に一元化

秘書コメントの役割: どこが気になるか / 何が確認できれば評価を変えられるか /
何の変化を示す材料か / 言えることと言えないことの境界 / 今後追うべき数字・発表・作品・興行・反応。
人格は「知ったかぶりをしないが、何を見るべきかは判断できる人」。

**人格定義は `docs/editorial-character.md` に書く**。プロンプトが同文書を毎回読み込む構造のため、
そこに書けば全生成工程（本文・コメント）へ自動で効き、二重管理を避けられる。
プロンプト側に人格記述を重複して持たない。

### D3. 事実台帳による生成工程の分離 — 2段階から開始し、第3段階はまず機械検査

工程: ① evidence から事実台帳を作る → ② 台帳だけを使って本文を書く → ③ 出力主張を台帳と照合。

台帳の claim 分類（4種、確定）:

- `verified_fact`: evidence に直接書かれている日付・数字・人物・出来事
- `source_analysis`: 元媒体による分析。必ず媒体名へ帰属
- `secretary_inference`: 秘書としての見方。判断であることを明示
- `unsupported`: 根拠を確認できない情報。出力禁止

Fable 検証での修正点:

- **③はまず機械検査（D5）のみで開始する**。LLM による再照合は偽陽性で出力本数が
  崩壊するリスクがあるため、機械検査で灰色になった記事と analysis_feature に限定して後付けする
- LLM 呼び出しは 1 topic あたり 1回 → 3〜4回（実行全体で約13回 → 35〜45回）になる。
  DeepSeek のコストは問題にならないが **JSON 切断の失敗面が増える**（F1 で実証済みの弱点）。
  台帳・本文とも `max_tokens` 明示と台帳の粒度上限（claim 最大件数）を必須とする
- 台帳のみから書くと文章が乾くため、D4 のコメント分離工程とセットで導入する

### D4. 秘書コメントは本文完成後の別工程で生成

本文とコメントの同時生成は事実と意見が混ざるため分離する。+1呼び出し/topic。

- コメント工程には**完成した本文を入力として渡し**、本文の言い換えを禁止する
  （`why_it_matters` と「ひとこと」の重複問題をここで同時に解く）
- コメントで禁止: 根拠のない「中国では一般的」/「大ヒット確実」等の断定 /
  evidence にない日本比較 / 新しい数字・人物名・作品名の追加 /
  「注目される」「期待される」だけの汎用コメント
- 「新しい固有名詞・数字の追加禁止」は D5 の機械検査（台帳照合）に落とす

### D5. 公開前の主張検査 — 硬いゲートと警告の二層に分ける

全項目をゲートにすると偽陽性で出力本数が崩れるため、二層とする。

| 検査 | 実装 | 扱い |
|---|---|---|
| 数字・日付・固有名詞が台帳に存在 | 文字列照合（中国語数詞・亿/万の正規化が必要） | **硬いゲート** |
| 事実文に evidence ID（claim ID）付与 | 台帳構造化で可能 | **硬いゲート** |
| 「確実」等の断定・汎用煽り表現 | 禁止語リスト | **硬いゲート** |
| 根拠のない日本比較 | 「日本」言及 × 台帳不一致で近似 | 硬いゲート（誤爆したら緩める） |
| 一媒体の分析を業界事実へ拡大 | 帰属表現の有無チェックで近似 | 警告（trace 記録のみ） |
| 原文と同じ段落順・論点順 | 文アライメントが必要で偽陽性多い | 警告（analysis_feature のみ将来ゲート化検討） |

導入手順: 検査は**最初 trace 記録のみ**で1〜2実行観察し、偽陽性率を見てからゲート化する。

実測で問題となった表現（禁止語リスト初版の種）:
「大ヒット確実」「これまで統一基準がなかった」「中高年層の音楽消費活性化」、
evidence にない日本の Inter BEE との比較。

### D6. 検査失敗時の劣化ポリシー（方針案に欠けていた点、必須）

いきなり記事ごと棄却しない段階制:

1. 違反文のみ削除して再検査
2. それでもダメなら再生成 **1回**
3. 棄却して backfill へ

加えて **1実行あたりの LLM 呼び出し総数上限**を設け、最悪時も出力が数本は残るようにする。

### D7. 実装は3分割し、各段で Actions 実測を挟む

- **Phase 3a**: 台帳抽出 + 台帳からの執筆（2段階化）。ハルシネーション・翻案対策の本体。
  本文 400〜700字への圧縮・元媒体分析への媒体名帰属・evidence 外の一般論禁止も
  このプロンプト再設計に含める
- **Phase 3b**: 秘書コメント分離（D4）+ 機械検査の硬いゲート（D5）+ 劣化ポリシー（D6）。
  検査は trace 記録のみ → 観察 → ゲート化の順
- **Phase 3c**: analysis_feature 解禁（D1）。editorial-character.md 改訂を含む

各段で AGENTS.md の実測ライン（官庁比率 ≦ 50% / 媒体 fresh > 0 / 最終本数・複数ソース topic 数の
維持）の後退がないことを確認してから次へ進む。

## 残項目の確定（Fable 設計セッション 2026-07-13 完了）

残項目1〜5はすべて確定した。詳細設計と Codex 実装指示は **`docs/design-phase3a-fact-ledger.md`** に記載。

1. **事実台帳 JSON スキーマ**: claims（C1〜最大20件、verified_fact / source_analysis / unsupported、
   evidence_refs・entities・numbers・quote_zh 付き）+ terms（用語解説、最大8件）+
   japan_availability（verified / not_in_evidence）+ unresolved。max_tokens 8000/8192。
   `secretary_inference` は 3b のコメント工程が追記する（台帳抽出では作らない）
2. **禁止語リストと線引き**: 3a の即時ゲートは「日本未公開断定」（全面禁止）と
   「予測断定（確実系）」の2種のみ。数字・固有名詞の台帳照合、無根拠一般論、日本比較、
   帰属なし分析、汎用コメントは warning（trace 記録のみ）→ 3b で観察後にゲート化判断
3. **劣化ポリシーと上限**: 違反文削除→再検査 → 執筆のみ再生成1回 → 棄却して backfill。
   LLM 呼び出しは 1実行45回（`LLM_CALL_BUDGET`）。超過時は新規 backfill / fallback を停止
4. **editorial-character.md 改訂文面**: 秘書人格・口調規定（少女口調は「秘書の注目ポイント」
   「秘書からのひとこと」の2セクションのみ）・用語初出解説・日本公開情報規定・帰属規定を
   3a で追記（文面は design-phase3a に確定記載）。analysis_feature 条件の追記は 3c で実施
5. **2-4（出力の再定義）は 3a に吸収**: publish_priority 順ソート + source_mix 表示 +
   official-only 低評価注記を renderMarkdown 改修に含める

追加で確定したユーザー要件（2026-07-13）:

- why_it_matters は「秘書の注目ポイント」へ改組（何を見るべきか・確認ポイント・追うべき数字）。
  「ひとこと」との重複削減を兼ねる
- 実害事例の再発防止: 给阿嬷的情书を「日本では未公開」と誤記（実際は面白映画配給で日本公開中）。
  日本公開・配信・字幕は台帳 japan_availability で確認できた場合のみ記載、否定断定は全面禁止
- 帰属の書き換え実例: 「過去の古偶一強から多様化へのシフトが鮮明」→
  「新浪エンタメは、古偶一強から多ジャンル化へ動いていると見ているみたいだよ！」
