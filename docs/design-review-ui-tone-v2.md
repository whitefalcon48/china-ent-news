# 設計: ローカルレビューUI ＋ ビンタン口調v2（Fable 設計セッション 2026-07-18）

GitHub Issue へ毎日判定コメントを書く運用（design-review-gate.md）は負担が大きく、毎朝5〜10分で完了できる形に改める。本設計は①レビュー判定をローカルWeb UI（`npm run review:ui`）で行う主運用への変更と、②ビンタンコメントの口調v2（丁寧語＋短いくだけた感想の混合）の2本立て。既存の review.json／review-apply ワークフロー／feedback JSONL／承認記事のみ公開の仕組みは**一切変更せず再利用**し、Issue コメント方式はフォールバックとして残す。本設計は design-review-gate.md の R-D1（Issue = レビューUI）と、design-phase3b-reader-first-quality.md の B1・B7-2・B8 の口調関連部分を supersede する。R1〜R7（レビューゲート）と B1〜B12（Phase 3b）は実装済み（2026-07-18・Actions実測待ち）のため、Codex は U1〜U7 を実装する。U6 は適用済みの B1・B7-2・B8 に対する差分適用（文面・検査の v2 更新）になる。

## 1. アーキテクチャ判断（UI-D1〜UI-D7）

- **UI-D1 UIは「コメント作成の視覚フロントエンド」**: ローカルUIは review.json を直接書き換えず、コミットも push もしない。UI 上の判定を design-review-gate §4 のコメント文法の1コメントに機械組み立てし、GitHub CLI（`gh issue comment`）で当日のレビュー Issue に投稿する。以降の処理（パース → review.json 更新 → feedback 蓄積 → 修正再生成 → 承認分のみ公開）は review-apply ワークフローがそのまま担う。判定の書き込み経路が Issue コメント1本に統一されるため、UI とスマホ（Issue 直接コメント）の併用でも整合が壊れない。
- **UI-D2 データ取得も GitHub CLI 経由**: `gh issue list --label daily-review --state open` で未完了レビュー日を特定し、`gh api repos/{owner}/{repo}/contents/...` で当該日の review.json と articles JSON を main ブランチから取得する。ローカル repo の pull は不要（GitHub 初心者でも git 操作ゼロで運用できる）。gh が使えない場合はローカル `data/` を読むフォールバック（古い可能性の警告を表示）。
- **UI-D3 ゼロ依存ローカルサーバ**: Node 標準の http のみで実装（サイトビルドと同じ zero-dep 方針）。バインドは 127.0.0.1 のみ、ポートは既定 4989（`REVIEW_UI_PORT` で変更可）。UI は外部 CDN を使わない自己完結 HTML 1枚。起動時に既定ブラウザを自動で開く（win32 は `start`、darwin は `open`、他は `xdg-open`）。
- **UI-D4 送信は最後に1回・冪等**: 送信確認画面で組み立て済みコメントをプレビューし、確定で1回だけ投稿する。同一番号への再判定は「後のコメントが上書き」という review-apply の既存規則（design-review-gate §4）でそのまま安全。送信失敗時はコメント全文のコピー UI と Issue への直リンクを表示し、手動貼り付けにフォールバックできる。
- **UI-D5 修正版の再判定対応**: review-apply が修正再生成後に articles JSON と review.json（revised_pending）を更新するため、UI は再起動時にそれを取得して「🔄 修正版」バッジ・revision_count・前回の修正指示付きで再表示する。判定操作は初回と同じ。
- **UI-D6 トークンを一切保持しない**: 認証は GitHub CLI に委譲する（`gh auth login` の資格情報は gh が Windows 資格情報マネージャーに保存）。トークンをリポジトリ・env ファイル・UI・localStorage のどこにも書かない。ブラウザは 127.0.0.1 のローカルサーバとしか通信せず、GitHub へのアクセスはすべてサーバ側の gh CLI 実行で行う。
- **UI-D7 Issue 方式はフォールバックとして維持**: design-review-gate R-D1 の「追加の Web アプリ・バックエンドは作らない」は本設計で supersede する（作るのは**ローカル閲覧・投稿専用サーバ**であり、公開バックエンドは引き続き作らない）。Issue 本文・コメント文法・review-apply は無変更のため、外出先はスマホから Issue に直接コメントすればよい。

## 2. 前提セットアップ（初回のみ・Windows / GitHub 初心者向け手順）

1. GitHub CLI のインストール: `winget install --id GitHub.cli`（PowerShell）
2. 新しいターミナルで `gh auth login` → GitHub.com → HTTPS → ブラウザでログイン（1回だけ）
3. 以後、毎朝は `npm run review:ui` の1コマンドのみ

`npm run review:ui` の preflight: gh 不在または未認証を検出したら、上記手順をそのまま表示して終了する（エラーで落とさない）。

仕様:

- 記事カード: バッジ・カテゴリ・確度チップ、タイトル、lead、what_happened（折りたたみ・既定は閉）、ビンタンコメント枠（常時表示）、ソースリンク、3ボタン
- 判定ボタンはトグル式。「修正」「却下」選択でタグチップ＋テキスト欄がインライン展開。却下の自由記述は任意、修正の指示は必須（プレースホルダ「短い修正指示（例: 口調が硬い。要するに構造で）」）。タグ既定値「その他」
- ヘッダに進捗（判定済み n/N＋バー）を常時表示。「残りをすべて採用」で未判定を一括採用
- 「🔄 修正版」バッジ: revised_pending の記事に表示。revision_count と前回の修正指示を添える
- キーボードショートカット: `j`/`↓` 次の記事、`k`/`↑` 前の記事、`a` 採用、`e` 修正、`x` 却下、`1`〜`6` タグ選択、`Enter` パネル確定、`z` 残りをすべて採用、`Ctrl+Enter` 送信確認へ
- 判定状態はブラウザ localStorage に自動保存（誤リロード対策。日付キー付き、送信成功後にクリア。判定内容のみでトークン等は含まない）
- 送信確認画面: 集計（採用N・修正N・却下N）＋組み立て済みコメントのプレビュー（読み取り専用＋コピー ボタン）→ [送信]。成功画面には Issue リンクと「修正がある場合は再生成後にもう一度 review:ui で再判定」の案内
- 未完了レビューが複数日ある場合はヘッダに日付セレクタ（既定は最新日）
- 操作時間の目安: 全採用なら `a`×10＋送信で1〜2分、修正・却下が数件でも5〜10分以内

## 4. GitHub 連携方法

| 操作 | 手段 |
|---|---|
| 未完了レビュー日の特定 | `gh issue list --label daily-review --state open --json number,title` |
| review.json / articles JSON 取得 | `gh api repos/{owner}/{repo}/contents/data/{date}/...`（main ブランチ） |
| 判定の送信 | `gh issue comment <issue_number> --body-file <一時ファイル>` |
| 以降の処理 | 既存 review-apply ワークフロー（無変更） |

- コメント組み立ては新規純関数 `buildReviewComment(decisions)` で行い、`parseReviewComment`（R4）との**ラウンドトリップ一致**（build→parse で同一判定に戻る）を受け入れ基準にする
- リポジトリの owner/repo は `gh repo view --json owner,name` またはローカル git remote から解決

## 5. セキュリティとフォールバック

- トークン非保持（UI-D6）。サーバは 127.0.0.1 バインドのみ・外部公開しない
- ブラウザからの POST はローカルサーバのみが受け、GitHub への書き込みは gh CLI 経由の Issue コメント1種類だけ（review.json 直接コミット等の書き込み経路を UI に持たせない）
- オーナー以外のコメント無視・author_association ガードは review-apply 側の既存仕様のまま（UI から投稿してもオーナー本人のコメントになる）

| 障害 | 挙動 |
|---|---|
| gh 未インストール／未認証 | セットアップ手順（§2）を表示して終了 |
| gh でのデータ取得失敗 | ローカル `data/` を読み「ローカルデータ（古い可能性）」の警告表示 |
| 送信失敗（ネットワーク・gh エラー） | コメント全文コピー UI＋Issue 直リンクを表示（手動貼り付けフォールバック） |
| 未完了レビューなし | 「今日のレビューはありません」を表示して終了 |
| ブラウザ自動起動失敗 | URL をターミナルに表示（手動で開く） |
| 外出などで PC が使えない日 | スマホから Issue に直接コメント（従来方式のまま有効） |

## 6. ビンタン口調v2（確定文面）

### 6-1. 変更の趣旨

です・ます調だけで統一すると固いため、丁寧語を土台に短いくだけた感想・リアクションを混ぜる。「かも」「みたい」は見方・可能性・伝聞専用とし、確認済み事実は言い切る規律は維持する。感嘆符の目安は2〜3個→**2〜4個**に広げ、0個は再生成対象にする。sober（重大事件・法的問題・訃報・被害者のいる話題）の0個は維持。

### 6-4. 執筆プロンプト（B7-1）は変更なし

執筆プロンプトのコメント関連指示は「docs/editorial-character.md の口調規定に従いビンタンの声で書く」という参照形式のため、editorial 文書の v2 差し替え（§6-2）だけで自動的に効く。B7-1 の文面は design-phase3b のまま使う（差し替え不要という判断を明記する）。

### 6-5. 機械検査の変更（design-phase3b B8 を次のとおり v2 に更新）

- `tone_exclamation`（v2）: sober かつ「！」>0 → gate（既存フローどおり再生成→文削除→sanitize）／ normal かつ「！」=0 → **再生成トリガ**（gate と同様にコメント再生成1回を発動する。ただし再生成後も0個の場合は文削除を行わず warning のまま採用する）／ normal の「！」>4 と1文2個以上は検査で弾かず `sanitizeExclamations` で決定的に修正し warning を記録
- `sanitizeExclamations`（v2）: sober → 全「！」「!」を「。」に置換 ／ normal → 1文内の2個目以降を「。」に置換し、**全体で5個目以降**を「。」に置換（2〜4個を許容）
- `template_comment` の正規表現に `今後注目したい` を追加
- `ending_repetition`（新規 warning）: コメント2欄合計で文末の「ですね。」「ですね！」が3回以上 → warning（再生成はしない。report:quality での観測用）
- `hedged_verified_fact`（v2で絞り込み）: 「かも」「みたい」「のようです」を含む文のうち、**台帳の numbers/entities に一致するトークンを含む文**（=確認済み事実を曖昧化している疑いが強い文）だけを warning にする（かも/みたいの見方用途を誤検知しないため）
- report:quality（B11）の「！」レンジ表示を 2〜4 に更新

## 7. 実装手順（U1〜U7。各ステップ後に `npm run check`）

- **U1** 起動スクリプトと preflight: `src/review/uiServer.ts`（新規・zero-dep）＋ package.json に `review:ui`。gh 存在・認証チェック（未認証時は §2 の手順を表示して終了）、`--dry-run` フラグ（ローカル data/ を読み、送信の代わりにコメント全文を画面表示）、ブラウザ自動起動、127.0.0.1 バインド・ポート 4989（`REVIEW_UI_PORT`）。受け入れ: gh 未認証で手順表示終了／`--dry-run` で UI が開く
- **U2** データ取得層: `src/review/fetchReviewData.ts`（新規）。gh issue list → 日付特定 → gh api で review.json＋articles JSON 取得、複数日はリスト返却、gh 失敗時ローカル data/ フォールバック＋警告フラグ。受け入れ: モックで「最新 pending 日の特定・複数日・フォールバック」の3ケース
- **U3** UI 画面本体: §3 の仕様どおり（カード・3ボタン・タグ・進捗・折りたたみ・ショートカット・localStorage 保存・日付セレクタ）。自己完結 HTML（CDN なし）。受け入れ: ダミーデータで §3 の全操作とショートカットが動く
- **U4** コメント組み立てと送信: `src/review/buildReviewComment.ts`（新規・純関数）＋ uiServer の POST /submit（gh issue comment 実行）。受け入れ: **build→parseReviewComment のラウンドトリップで判定が一致**（採用/修正/却下/タグ省略/残り採用）／送信失敗時にコピー UI と Issue リンクが出る
- **U5** 修正版の再判定対応: revised_pending の「🔄 修正版」バッジ・revision_count・前回指示の表示。受け入れ: ダミー review.json（revised_pending 混在）で表示と再判定が動く
- **U6** 口調v2の適用: §6-2 の editorial-character.md 差し替え（一字どおり）、§6-3 の実装済み `buildBingtangCommentPrompt` の差し替え、§6-5 の実装済み `runCommentCheck` / `sanitizeExclamations` の検査変更。受け入れ: 差分が §6-2 と一致／ダミーで「normal ！=0 → 再生成トリガ」「！5個 → sanitize で4個」「ですね×3 → warning」「かも＋台帳数値 → warning、かも単独 → 検出なし」が通る

## 8. ローカル検証手順

1. 各ステップ後 `npm run check`
2. `npm run review:ui -- --dry-run` で、ダミー review.json＋articles JSON（scratchpad に配置・コミットしない）から: 起動→自動でブラウザが開く→10本を `a`/`e`/`x`＋タグで判定→進捗表示→送信確認画面のコメントプレビューが文法どおり→dry-run 出力を parseReviewComment に通して判定一致
3. localStorage 復元: 判定途中でリロード→状態が残る／送信成功後→クリア
4. 送信失敗パス: gh を PATH から外して送信→コピー UI と Issue リンクが表示される
5. 修正版: revised_pending 混在ダミーで 🔄 バッジと前回指示の表示を確認
6. 口調v2 検査: U6 受け入れのダミーケース一式
7. ストップウォッチ検証: ダミー10本を「採用8・修正1・却下1」で判定して送信まで **10分以内**（目標5分）

## 9. Actions での受け入れ基準

1. UI から送信したコメントで review-apply が従来どおり動く（review.json 更新・feedback 蓄積・修正再生成・全確定で承認分のみ公開・Issue クローズ）
2. スマホから Issue へ直接コメントした場合も同一挙動（フォールバック維持）
3. 修正再生成後、UI 再起動で修正版が再判定でき、再送信で確定する
4. 口調v2（U6 適用後の Actions 実測）: 通常記事のコメント欄「！」2〜4個・1文1個以下／「！」0個の記事はコメント再生成が trace に記録される／sober 記事は「！」0個／「かも」「みたい」が確認済み事実に付かない／「ですね」文末が1記事のコメント欄に3回以上出ない／「今後注目したい」型の締めが出ない／くだけた感想（「すごい！」等）が過剰でない（1コメント欄1回まで）
5. トークンがリポジトリ・ログ・trace のどこにも現れない
6. 維持ライン（官庁比率 ≦50%・媒体 fresh >0・複数ソース topic 数）が後退しない

## 10. やらないこと

- review.json への直接書き込み・ローカルからの git push（書き込みは Issue コメント経由のみ）
- 公開 Web アプリ・認証付きバックエンド・トークンの独自管理
- review-apply／コメント文法／feedback スキーマの変更（完全再利用）
- Issue 方式の廃止（フォールバックとして維持）
- Phase 3c・Phase 4 サイト UI への変更
